5 — Review Assessment
✓ Strengths
  • Additive-only approach — zero existing behavior modified
  • Clear dependency chain — each phase blocks the next
  • Service/store patterns already established — Phase 02 is low-risk cloning
  • Backend endpoint is simple clone — remove one filter, reuse everything else
  • TimesheetGrid + CapacityDashboard already have cross-workspace toggles
  • Rollback story is clean: delete new files + route entries
⚠ Concerns
  • TypeScript: making projectId optional may break existing callers — need to verify
  • AnalyticsTimesheetTable issue link construction may depend on projectId
  • Route registration order matters — Phase 05 must run after Phase 04 creates files
  • react-router type file auto-generation is timing-dependent
  • No pagination on new workspace endpoint — large workspaces could be slow
✗ Gaps
  • Plan doesn't specify workspace-level permission check for new endpoint
  • No mention of workspace member vs admin distinction in timesheet visibility
  • WeekStart parameter validation not detailed — what happens with bad dates?
  • No test coverage plan for the new endpoint
Understanding Gaps
  • ?  Does the new workspace analytics endpoint need explicit ADMIN permission or does middleware handle workspace-level auth?
  • ?  Are there existing workspace-level timesheet permission rules that differ from project-level?
  • ?  Should the workspace time-tracking pages have a feature flag guard like project pages do?
  • ?  Should time-tracking be accessible to all workspace members or ADMIN-only at workspace level?

  • /:ws/time-tracking/  renders Timesheet tab with cross-workspace data
  • /:ws/time-tracking/analytics  renders workspace-scoped analytics
  • /:ws/time-tracking/capacity  renders capacity heatmap cross-workspace
  • Sidebar shows "Time Tracking" nav item with Timer icon
  • No regression on project-level time-tracking pages
  • GET /api/workspaces/{slug}/time-tracking/analytics/timesheet/ returns 200
  • Permission gated to ADMIN + MEMBER
  • No N+1 queries — select_related used
